# 亲子识字 H5 生成搜字迭代设计稿

日期：2026-04-25

## 1. 项目概述

MVP 已经完成并发布，下一轮迭代不再只是扩充少量精选字卡，而是把首页“搜一个字”升级成可生成入口。

本轮设计的目标不是开放任意汉字实时生成，而是在明确边界内，验证“用户搜索一个白名单内的字时，系统能在一次请求内生成一张可直接带孩子学的文本卡片”。

本轮已经确认的关键决策如下：

- 目标方向是 `扩内容和找字能力`
- 首版只做 `纯文本生成卡`
- 只覆盖 `300-500` 个高频基础字
- 白名单外明确不支持，不做长尾兜底
- 采用 `方案 B：白名单 + 结构化生成 + 私有缓存首发`
- 首次命中时再生成，生成成功后缓存复用
- 质量优先，允许同次请求内自动重试
- 首次等待时间目标为 `10-20 秒`
- 生成成功后先只对当前搜索用户可用，后续再决定是否提升为公共缓存
- 部署形态允许引入 `轻量 Serverless / Edge`，不引入传统常驻服务器

## 2. 产品目标与非目标

### 2.1 本轮目标

- 让首页搜索从“只命中本地精选卡”升级为“白名单内可生成可学习的字卡”
- 保持现有学习页结构不变，继续围绕同一套 `HanziCard` 文本字段工作
- 用最小后端能力验证生成卡是否真的能被家长带完一次 `5 分钟` 共学流程
- 为后续公共缓存、人工审核、白名单扩张和提示词迭代打下数据基础

### 2.2 明确不做

- 不支持白名单外任意汉字生成
- 不生成图片、插图或更自然的语音资源
- 不做账号体系和跨设备同步
- 不做完整运营 CMS，只做最小可用的生成结果管理能力
- 不把所有生成成功的卡直接公开给所有用户

## 3. 用户与成功标准

### 3.1 目标用户

- 3-6 岁儿童家庭中的家长
- 已经接受当前 H5 是“轻量陪学工具”，而不是完整课程产品

### 3.2 关键用户故事

- 家长在首页输入一个常见汉字，希望快速得到一张可直接带孩子学的字卡
- 当这个字已经准备好时，家长应当直接进入学习页，而不是再经历复杂确认
- 当这个字首次被搜索时，家长可以接受明显等待，但结果必须足够好
- 当这个字暂不支持或生成失败时，系统要明确说明，而不是模糊提示“没找到”

### 3.3 成功标准

- 白名单内搜索字的首搜生成成功率可被稳定追踪
- 生成成功的卡片能直接进入现有学习页，无需为生成卡单独造一套 UI
- 白名单外与生成失败的反馈清晰，用户不会误以为系统故障
- 后台能够查看生成结果、失败原因和后续是否转为公共缓存

## 4. 产品行为设计

### 4.1 首页搜索主链路

首页搜索在本轮变成两段式解析：

1. 前端先查本地精选卡
2. 如果命中，直接进入现有精选卡学习页
3. 如果未命中，调用远端解析接口检查白名单和缓存状态
4. 白名单外，返回“首版暂不支持这个字”
5. 白名单内且已有可用缓存，直接返回对应卡片
6. 白名单内但无可用缓存，进入生成流程
7. 生成通过后立即让当前用户进入学习页
8. 生成失败则返回“这个字还没准备好”

### 4.2 状态模型

后端需要维护以下业务状态：

- `curated`
  现有人工精选卡，本地即可命中
- `unsupported`
  不在本轮 `300-500` 白名单内
- `ready_private`
  已生成成功，但仅当前搜索用户可访问
- `ready_public`
  后续被提升为公共缓存，所有用户都可访问
- `failed`
  在本次生成窗口内仍未通过质量门槛

`generating` 不作为长期存储状态暴露给产品层，它是前端在调用生成接口期间展示的瞬时 UI 状态。

### 4.3 前端体验要求

- 首页搜索反馈不再统一显示“没找到这个字卡”
- 至少需要区分：
  - `暂不支持这个字`
  - `正在为你准备`
  - `这个字还没准备好`
- 生成卡与精选卡共用学习页结构，不单独设计第二套详情页
- “精选字卡列表”继续只展示人工精选卡，不自动混入私有生成卡
- “我的”页面可记录用户访问过的生成卡，但仅在当前匿名身份有效

## 5. 数据契约与标识

### 5.1 复用现有 `HanziCard`

本轮不改学习页消费的数据结构，继续复用当前 `HanziCard`：

- `slug`
- `character`
- `pinyin`
- `theme`
- `estimatedMinutes`
- `heroLine`
- `storyScene`
- `storyText`
- `parentPrompt`
- `words`
- `sentences`
- `activityPrompt`

其中生成卡也必须输出同一结构，避免前端按卡片来源分叉渲染逻辑。

### 5.2 生成卡字段约束

为保证稳定性，生成卡需要遵守额外约束：

- `character` 必须等于用户搜索字
- `pinyin` 延续当前格式，使用不带声调的小写拼音
- `estimatedMinutes` 固定为 `5`
- `words` 固定输出 `3` 个
- `sentences` 固定输出 `2` 个
- 所有文本字段必须是纯文本，不带 markdown、编号或解释性前缀

### 5.3 匿名用户标识

首版不引入登录，但必须引入浏览器级匿名身份标识：

- 前端首次访问时生成一个稳定的匿名 `browser_id`
- 存在本地持久化存储中
- 所有远端解析、取卡、生成请求都带上这个标识

这样后端才能正确隔离 `ready_private` 卡片的访问权限。

副作用是：

- 用户清空本地存储后，之前的私有生成卡不可继续访问
- 不支持跨设备同步

这两点在本轮范围内可以接受。

### 5.4 卡片标识与路由约定

为避免前端同时维护两套详情页路由，卡片访问统一通过远端返回的 `cardId` 进入：

- 精选卡继续保留现有 `slug`
- 生成卡也必须拥有可路由的稳定 `cardId`
- 前端搜索完成后，不直接自行拼接生成卡路由，而是使用解析接口返回的目标标识

首版建议的约定是：

- 精选卡：仍可用既有 `slug`
- 私有生成卡：使用独立 `cardId`，不把 `browser_id` 直接暴露到 URL
- 公共生成卡：在被提升为公共缓存后，也保留独立 `cardId`

这样前端只需要知道“这是一个可打开的卡片标识”，不用自己推断其来源和权限。

## 6. 系统架构设计

### 6.1 静态前端

前端继续作为静态站点部署。职责只有：

- 搜索输入与状态反馈
- 管理匿名 `browser_id`
- 本地精选卡命中
- 拉取远端卡片数据
- 渲染学习页与本地进度

前端不直接接模型，也不承担生成逻辑。

### 6.2 Serverless API

远端服务建议至少包含三类接口：

- `POST /api/search/resolve`
  输入：`browser_id`、搜索字
  输出：`unsupported / ready_private / ready_public / needs_generation`
- `POST /api/generate`
  输入：`browser_id`、搜索字
  行为：执行一次完整生成流程并返回结果
- `GET /api/cards/:cardId`
  输入：`browser_id`
  行为：按权限返回公共卡或私有卡的完整 `HanziCard`

前端的典型顺序是：

1. 本地未命中后调用 `search/resolve`
2. 若返回 `needs_generation`，前端展示等待态并调用 `generate`
3. 成功后拿到 `cardId`，进入学习页
4. 学习页再按 `cardId` 拉完整卡片

### 6.3 生成编排器

`generate` 不能是“调用一次模型然后直接落库”的裸流程，必须有生成编排器，负责：

- 选择固定提示词模板
- 用固定 schema 请求结构化输出
- 做结构校验
- 做规则校验
- 做内容审校
- 在同次请求内控制重试
- 记录失败原因和耗时
- 通过后写入私有缓存

### 6.4 缓存与运营层

至少需要以下几类持久化对象：

- `whitelist_chars`
  记录本轮允许搜索的 `300-500` 个字及启用状态
- `private_cards`
  仅对 `browser_id` 可见的生成卡
- `public_cards`
  已被提升为公共缓存的生成卡
- `generation_jobs`
  记录每次生成的输入、尝试次数、耗时、失败原因、提示词版本、模型版本

公共缓存和私有缓存必须物理分离，不只靠一个状态字段混存。这样权限判断、运营提升和统计都会更清晰。

### 6.5 最小运营后台

本轮不做完整 CMS，但需要最小管理能力：

- 浏览最近生成成功的私有卡
- 浏览失败记录
- 查看某个字的最近生成结果
- 将私有卡提升为公共缓存
- 调整白名单启用状态

## 7. 生成与质量门槛

### 7.1 质量控制原则

本轮的核心不是“能生成”，而是“生成后能直接支撑一次亲子共学”。因此质量控制要拆成三层。

### 7.2 第一层：结构门槛

结构校验由代码完成，要求：

- 返回结果完整符合 `HanziCard`
- `character` 与搜索字一致
- `words` 数量为 `3`
- `sentences` 数量为 `2`
- 各字段不能为空
- 文本长度处于合理范围，避免过短或失控过长

### 7.3 第二层：规则门槛

规则校验由代码完成，要求：

- 词和句中至少部分直接包含目标字
- 不允许重复词、重复句
- 句子长度适合 `3-6` 岁亲子共读
- `parentPrompt` 必须是家长带读提示，而不是百科解释
- `activityPrompt` 必须能立即执行，不依赖额外教具

### 7.4 第三层：内容门槛

内容门槛可由单独审校步骤完成，重点检查：

- 故事是否自然，避免写成字典释义
- 内容是否真正围绕目标字
- 文案是否能支撑家长带完约 `5` 分钟流程
- 语气是否符合亲子共学，而不是课堂讲义

审校结果应返回结构化结论，而不是只给自由文本，以便驱动自动重试。

### 7.5 重试策略

在用户已接受 `10-20 秒` 等待的前提下，首版重试上限建议为 `3` 次机会：

1. 初稿生成
2. 基于失败原因的定向修复
3. 完整重生一次

如果三次机会后仍未通过质量门槛，则本次生成失败。

本轮不做“后台慢慢生成、稍后再看”的异步模式，失败就明确返回不可用。

## 8. 权限、异常与边界情况

### 8.1 私有卡权限

- `ready_private` 卡只能由拥有对应 `browser_id` 的请求获取
- 不能仅凭 `slug` 或 `cardId` 直接公开访问
- 同一个字在被提升为公共缓存前，不做跨用户复用

这意味着不同用户在公共化前可能为同一个字分别触发生成，这是本轮为保证权限清晰而接受的成本。

### 8.2 异常处理

- 白名单外：直接返回 `unsupported`
- 生成超时或三次失败：返回 `failed`
- 网络错误：前端提示重试，不伪装成“不支持”
- 身份丢失：若用户清空本地标识，之前私有卡不可再访问

### 8.3 并发与重复请求

同一用户对同一字的重复请求需要具备幂等性：

- 如果已有私有卡，直接命中
- 如果当前正在生成，前端应复用同一次请求结果，避免重复扣费和重复写入

## 9. 测试与埋点

### 9.1 必要测试

- 单元测试
  - 白名单判断
  - 匿名身份生成与读取
  - 私有 / 公共权限判断
  - 状态映射与缓存读写
- 合约测试
  - API 返回的 `HanziCard` schema
- 生成回归测试
  - 对一组代表字执行固定样例校验，防止提示词或模型调整后质量回退
- 前端集成测试
  - 搜索后进入 `unsupported / generating / ready_private / failed` 的不同状态

### 9.2 首版必须记录的指标

- 白名单命中率
- 首搜生成成功率
- 平均重试次数
- `P50 / P95` 生成耗时
- 失败原因分布
- 私有卡提升为公共缓存的转化率
- 白名单外搜索次数

这些指标将直接决定下一轮是扩白名单、补运营审核，还是继续优化生成链路。

## 10. 迭代边界与后续方向

本轮落地后，系统能力边界会变成：

- 精选卡继续作为稳定入口
- 白名单内支持按需生成文本卡
- 生成成功后先私有可用，再人工决定是否公共化

下一阶段再根据数据决定是否进入以下方向：

- 扩大白名单范围
- 引入“热字预生成”
- 为生成卡增加插图或更强音频
- 增加更完整的审核后台

当前阶段不提前承诺这些扩展，先验证“高质量白名单生成搜字”是否成立。
